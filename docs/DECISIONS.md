# Decisions

Append-only log of judgment calls not dictated by [SPEC.md](./SPEC.md). One line
each: date — decision — why. (Per SPEC.md §2.7.)

- 2026-07-09 — Package manager is pnpm (via corepack), per SPEC.md Phase 5 criteria which references `pnpm dev`. — Spec assumes pnpm.
- 2026-07-09 — Scaffolded with `create-next-app` giving Next.js 16.2 / React 19 (spec says "Next.js 15"). — Latest stable at build time; App Router API used is compatible with the spec's intent. Revisit only if a spec-required behavior breaks.
- 2026-07-09 — MapLibre is mounted client-only through a small `MapLoader` wrapper because App Router disallows `ssr:false` `next/dynamic` inside Server Components. — Framework constraint; keeps the map out of SSR where it cannot run.
- 2026-07-09 — "The Survey" base-map desaturation is done by transforming the Liberty **style JSON** (`components/map/mapStyle.ts`), not by a CSS canvas filter. — A CSS `saturate()` filter over the whole canvas also washes the oceans to near-white, violating SPEC.md §7 "keep water recognizably blue." The style transform desaturates land while explicitly recoloring water to a hydrographic blue.
- 2026-07-09 — Water fill recolored to a muted pale blue (hue 210, low saturation) rather than the full-strength `--water` (#3B6FA0) token. — #3B6FA0 as an ocean fill is too heavy for a chart-paper aesthetic; the `--water` token is reserved for links/active UI and water *labels*. Ocean saturation is a tunable and may be revisited during Phase 5 polish.
- 2026-07-10 — Auth uses cloud Supabase, not a local Docker stack. — User is new to Docker and the app is already on Vercel (which cannot reach a local DB); cloud matches prod, keeps migrations as the source of truth, and avoids disk pressure. Migrations are still the source of truth (pushed via `supabase db push --db-url`), so a contributor can later run the same files against a local stack.
- 2026-07-10 — OSM identity is resolved server-side from `floor(feature.id/10)` + an Overpass node/way/relation lookup, not decoded from the tile feature id's type code. — SPEC.md §4.1 assumed tiles expose OSM ids as properties; verification showed OpenFreeMap/Planetiler tiles only carry an encoded id in `feature.id` whose trailing type code is unreliable. See ARCHITECTURE.md "OSM feature identity from tiles."
- 2026-07-10 — Phase 2 stores point identity + kind + centroid only; boundary polygons, ancestry, and resolving place-label nodes to their admin relation are deferred to Phase 3. — Phase 2's acceptance criteria only require a correct `places` row (name/kind/coordinates) per tap; polygons are needed for containment and the parcel outline, which are Phase 3.
- 2026-07-10 — Map label balance tuned in the style transform: bus stops dropped from the always-on transit layer, a park label layer added (Liberty ships none), neighborhoods shown a touch earlier, street/boulevard name labels held to z15. — User feedback: surface clickable containers (neighborhoods, parks) and hide un-clickable transit/street noise when zoomed out.
- 2026-07-10 — Map vibe changed from "The Survey" (desaturated chart-paper) to "Modern Vivid" (bright, mostly true-to-life color, near-white ground, magenta selection kept). The transform is now theme-driven (`MapTheme` in mapStyle.ts), so the vibe is a one-object swap. — User asked to change the map's color/vibe and chose Modern Vivid; SPEC.md §7's "Survey" aesthetic was the author's original call but the product owner elected a brighter look. Other vibes (Night Chart, Vintage Atlas, Blueprint) are documented as alternatives if revisited.
- 2026-07-10 — Chat upgraded from one flat chat per place to a Slack-style place workspace: channels (#general/#q-and-a/#events + custom), place-scoped DM threads, threaded replies, and reactions. Interaction patterns are Slack's; the skin stays ink-on-paper. — Product direction (dedicated chat-UX prompt). Existing flat chats backfill into each place's #general.
- 2026-07-10 — Default channels are created by an AFTER INSERT trigger on `places` (not only inside `hydrate_place`), so a place gets its three channels no matter how it's created (OSM hydration or a user's custom pin). — The invariant "a place always has its default channels" must hold across both insert paths; a single trigger guarantees it.
- 2026-07-10 — DMs are private: the previously blanket `messages` SELECT policy (`using(true)`) was replaced so channel/legacy messages stay public but DM messages are readable only by thread participants; DM readers are SECURITY DEFINER with an explicit membership check. — The old policy would have exposed DM content to everyone, including via Realtime. Channel content stays fully public (the map is public).
- 2026-07-10 — The place sheet widens to 600px on desktop with the channel rail beside the stream; on mobile it's a push stack (rail → stream → thread) rather than a cramped side-by-side. — A persistent Slack 3-pane doesn't fit the 400px sheet; this matches the existing mobile-bottom-sheet / desktop-right-panel responsive behavior.
- 2026-07-10 — Messages and reactions subscribe on SEPARATE Supabase Realtime channels. — Empirically, putting two postgres_changes bindings of different filter shapes (messages filtered by parent_id + reactions unfiltered) on one channel makes Supabase silently drop delivery for all of them; splitting keeps each robust. Verified both deliver in ~1.3s.
- 2026-07-10 — Reactions realtime is a global stream (no server-side filter, since reactions carry no channel column), limited to channel-message reactions by RLS and filtered to visible messages client-side. — Acceptable for MVP volume; a denormalized parent column for server-side filtering is a scaling fast-follow. DM reactions are excluded from the realtime policy and only reach participants via the DM reader.
- 2026-07-10 — No breadcrumb in the rail (the breadcrumb/containment feature was cut from the MVP); place identity (name/kind/coords) stays in the sheet header above the workspace. — The spec assumed a breadcrumb existed; it doesn't yet.
