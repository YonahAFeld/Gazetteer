# Decisions

Append-only log of judgment calls not dictated by [SPEC.md](./SPEC.md). One line
each: date — decision — why. (Per SPEC.md §2.7.)

- 2026-07-09 — Package manager is pnpm (via corepack), per SPEC.md Phase 5 criteria which references `pnpm dev`. — Spec assumes pnpm.
- 2026-07-09 — Scaffolded with `create-next-app` giving Next.js 16.2 / React 19 (spec says "Next.js 15"). — Latest stable at build time; App Router API used is compatible with the spec's intent. Revisit only if a spec-required behavior breaks.
- 2026-07-09 — MapLibre is mounted client-only through a small `MapLoader` wrapper because App Router disallows `ssr:false` `next/dynamic` inside Server Components. — Framework constraint; keeps the map out of SSR where it cannot run.
- 2026-07-09 — "The Survey" base-map desaturation is done by transforming the Liberty **style JSON** (`components/map/mapStyle.ts`), not by a CSS canvas filter. — A CSS `saturate()` filter over the whole canvas also washes the oceans to near-white, violating SPEC.md §7 "keep water recognizably blue." The style transform desaturates land while explicitly recoloring water to a hydrographic blue.
- 2026-07-09 — Water fill recolored to a muted pale blue (hue 210, low saturation) rather than the full-strength `--water` (#3B6FA0) token. — #3B6FA0 as an ocean fill is too heavy for a chart-paper aesthetic; the `--water` token is reserved for links/active UI and water *labels*. Ocean saturation is a tunable and may be revisited during Phase 5 polish.
- 2026-07-10 — Auth uses cloud Supabase, not a local Docker stack. — User is new to Docker and the app is already on Vercel (which cannot reach a local DB); cloud matches prod, keeps migrations as the source of truth, and avoids disk pressure. Migrations are still the source of truth (pushed via `supabase db push --db-url`), so a contributor can later run the same files against a local stack.
- 2026-07-10 — OSM identity is resolved server-side from `floor(feature.id/10)` + an Overpass node/way/relation lookup, not decoded from the tile feature id's type code. — SPEC.md §4.1 assumed tiles expose OSM ids as properties; verification showed OpenFreeMap/Planetiler tiles only carry an encoded id in `feature.id` whose trailing type code is unreliable. See ARCHITECTURE.md "OSM feature identity from tiles."
- 2026-07-10 — Phase 2 stores point identity + kind + centroid only; boundary polygons, ancestry, and resolving place-label nodes to their admin relation are deferred to Phase 3. — Phase 2's acceptance criteria only require a correct `places` row (name/kind/coordinates) per tap; polygons are needed for containment and the parcel outline, which are Phase 3.
- 2026-07-10 — Map label balance tuned in the style transform: bus stops dropped from the always-on transit layer, a park label layer added (Liberty ships none), neighborhoods shown a touch earlier, street/boulevard name labels held to z15. — User feedback: surface clickable containers (neighborhoods, parks) and hide un-clickable transit/street noise when zoomed out.
- 2026-07-10 — Map vibe changed from "The Survey" (desaturated chart-paper) to "Modern Vivid" (bright, mostly true-to-life color, near-white ground, magenta selection kept). The transform is now theme-driven (`MapTheme` in mapStyle.ts), so the vibe is a one-object swap. — User asked to change the map's color/vibe and chose Modern Vivid; SPEC.md §7's "Survey" aesthetic was the author's original call but the product owner elected a brighter look. Other vibes (Night Chart, Vintage Atlas, Blueprint) are documented as alternatives if revisited.
- 2026-07-10 — Chat upgraded from one flat chat per place to a Slack-style place workspace: channels (#general/#q-and-a/#events + custom), place-scoped DM threads, threaded replies, and reactions. Interaction patterns are Slack's; the skin stays ink-on-paper. — Product direction (dedicated chat-UX prompt). Existing flat chats backfill into each place's #general.
- 2026-07-10 — Default channels are created by an AFTER INSERT trigger on `places` (not only inside `hydrate_place`), so a place gets its three channels no matter how it's created (OSM hydration or a user's custom pin). — The invariant "a place always has its default channels" must hold across both insert paths; a single trigger guarantees it.
- 2026-07-10 — DMs are private: the previously blanket `messages` SELECT policy (`using(true)`) was replaced so channel/legacy messages stay public but DM messages are readable only by thread participants; DM readers are SECURITY DEFINER with an explicit membership check. — The old policy would have exposed DM content to everyone, including via Realtime. Channel content stays fully public (the map is public).
- 2026-07-10 — The place sheet widens to 600px on desktop with the channel rail beside the stream; on mobile it's a push stack (rail → stream → thread) rather than a cramped side-by-side. — A persistent Slack 3-pane doesn't fit the 400px sheet; this matches the existing mobile-bottom-sheet / desktop-right-panel responsive behavior.
- 2026-07-10 — Messages and reactions subscribe on SEPARATE Supabase Realtime channels. — Empirically, putting two postgres_changes bindings of different filter shapes (messages filtered by parent_id + reactions unfiltered) on one channel makes Supabase silently drop delivery for all of them; splitting keeps each robust. Verified both deliver in ~1.3s.
- 2026-07-10 — Reactions realtime is a global stream (no server-side filter, since reactions carry no channel column), limited to channel-message reactions by RLS and filtered to visible messages client-side. — Acceptable for MVP volume; a denormalized parent column for server-side filtering is a scaling fast-follow. DM reactions are excluded from the realtime policy and only reach participants via the DM reader.
- 2026-07-10 — No breadcrumb in the rail (the breadcrumb/containment feature was cut from the MVP); place identity (name/kind/coords) stays in the sheet header above the workspace. — The spec assumed a breadcrumb existed; it doesn't yet.
- 2026-07-10 — Initial map view uses one-shot browser geolocation (6s timeout, 5min position cache) to center on the user at zoom 13 ("neighborhood/city" scale); falls back to the existing world view (center [0,20], zoom 2) if denied, unavailable, or slow. Requested in parallel with the style fetch so it doesn't add latency to first paint. — User wanted the map to open somewhere relevant instead of the whole planet; the existing GeolocateControl button still lets a user re-center manually at any time.
- 2026-07-10 — `lib/supabase/client.ts` no longer crashes `next build` when Supabase env vars are missing. Root cause found in production: `AccountControl` (a Client Component) constructs the browser Supabase client during render, which Next.js also executes once in Node during `next build`'s static prerender of "/" — with envs absent, `@supabase/ssr` threw synchronously there, failing the entire build (Vercel then silently kept serving the last successful deployment). The prerender pass never actually calls Supabase (effects don't run during it), so `createClient()` now hands that Node-only, no-window pass harmless placeholder credentials instead of crashing; a real browser with the same misconfiguration still throws immediately, so it fails loud in that user's console/error boundary rather than silently failing the whole deployment. Verified by reproducing the crash with env vars unset (pre-fix) and confirming a clean build with the same envs unset (post-fix).
- 2026-07-10 — Added two onboarding aids since the product's premise ("every place is a room") wasn't self-evident on first load: a persistent bottom-left "?" control (`components/onboarding/InfoControl.tsx`) with a short About + "How it works" panel, and a one-time top-center hint (`components/onboarding/FirstVisitHint.tsx`) nudging the core tap-a-place interaction — dismissible, persisted via localStorage, and auto-dismissed the moment the user actually selects something. Both are non-modal overlays, matching the "never a modal wall" principle already established for the composer's sign-in nudge.
- 2026-07-10 — Set `devIndicators.position: "top-left"` in next.config.ts (dev-only, no effect on production). Found via headless-browser verification that Next.js's own dev-mode indicator badge defaults to the exact same bottom-left corner as the new InfoControl button and was rendering on top of it, intercepting clicks locally. Top-left is the one corner nothing else in the app anchors to (top-right has sign-in, bottom-right has the map's zoom/geolocate controls).
- 2026-07-10 — Search reuses the existing hydrate pipeline unchanged: Nominatim already resolves osm_type via its own geocoding, but `hydratePlace`'s Overpass resolver (`resolveOsmElement`) re-derives type from a bare numeric id anyway (querying node/way/relation and disambiguating by name), so a search result only needs to send `{osmId, name, lng, lat}` — the exact optional-osmId path `/api/geo/hydrate` already supported for non-tap callers. No backend changes needed beyond the new `/api/geo/search` proxy.
- 2026-07-10 — `/api/geo/search` applies two separate rate limits: a generous per-client-IP one (abuse protection for our route) and a strict global one keyed by a constant string (Nominatim's usage policy caps a single instance at 1 req/sec regardless of caller). Both reuse the existing `lib/rateLimit.ts` token bucket.
- 2026-07-10 — MapCanvas gained a `flyTo` prop (lng/lat/token) so search — the one interaction that jumps to a place not currently on screen — can move the camera; ordinary taps never needed this since the tapped place is already in view. Keyed by a caller-incremented `token` so re-selecting the same result still flies.
- 2026-07-10 — Replaced the `/`-triggered search overlay with a persistent, always-visible pill search bar glued to top-center (Google Maps convention), per explicit user request. This is a scoped, intentional exception to "The Survey"'s flat 0–2px-radius rule (rounded-3xl pill + soft drop shadow, not the flat ink-offset shadow used elsewhere) — the map already has a rounded-pill precedent in the clickable place chips, and the user asked for the Google Maps look specifically. `/` now focuses the always-present input rather than toggling a modal-like overlay's visibility; component manages its own focused/query state internally (MapView no longer needs searchOpen state). FirstVisitHint moved from top-3 to top-20 so it stacks below the bar instead of overlapping it.
- 2026-07-11 — Fixed the "message disappeared" bug: OSM models one real place (e.g. Pembroke Pines) as multiple elements — an administrative boundary relation AND a separate label node — and Gazetteer's per-osm-element identity split them into two places with two separate (one empty) chats. Confirmed via direct DB query: the user's message was safe in the relation-based row; a later tap/search resolved the node-based row instead. Fixed server-side with a new `place_by_name_near` dedup reader (migration 20260711033000): before minting a new place, hydration now checks for an existing place of the same kind+name within a kind-scaled radius (300km for countries down to 100m for buildings; disabled for custom pins) and treats a match as an alias instead of splitting the chat. The one existing empty duplicate (confirmed zero messages/DMs/ancestry) was deleted directly.
- 2026-07-11 — Chip-highlight fallback: the map's "which chip is selected" reconciliation in MapCanvas.tsx required an *exact* osm_id match against the rendered tile feature. Since a canonical place's stored osm_type/osm_id can differ from whichever OSM element Planetiler chose to render the on-screen label (same node-vs-relation duplicate-identity issue as above), this could never highlight the pill for e.g. a search-selected place. Added a name-based fallback (case-insensitive) among currently-rendered chip-eligible features when the exact match fails.
- 2026-07-11 — Search results now carry a `zoom` hint derived from Nominatim's `type` field (country≈4 … neighbourhood≈14, default 15) instead of the flyTo hardcoding `max(current zoom, 14)`. A city's label chip doesn't render at street-level zoom, so if the map was already zoomed in before a search, flying to a city result at an unchanged too-close zoom left no chip on screen at all — not even the dot lined up with a real target. `FlyToTarget` now carries an explicit `zoom` instead of MapCanvas guessing one.
- 2026-07-11 — Chip-reconciliation on a flyTo (search) now re-runs on the map's `idle` event (plus one extra retry ~600ms later), not `moveend` — moveend only signals the camera animation finished, but the destination's vector tiles can still be loading, so querying rendered features right at moveend can still only see the pre-flight tiles. NOTE: verification of the on-screen highlight itself was inconclusive in this project's headless SwiftShader test harness (queryRenderedFeatures() kept returning stale world-zoom content there even after confirming, via separate debug logs, that idle fired with the correct destination zoom/center — while a plain screenshot of the same flow correctly rendered full destination-area tile detail). This looks like a quirk of driving queryRenderedFeatures() through CDP's Runtime.evaluate in software rendering, not a reproducible app bug, but it means the visual highlight for search-selected places specifically has not been confirmed with certainty — worth a real-browser check.
